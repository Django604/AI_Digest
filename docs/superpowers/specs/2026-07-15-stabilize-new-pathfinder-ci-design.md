# 稳定新探陆 CI 测试设计

## 问题

新探陆功能测试直接断言生产工作簿在 2026 年 7 月 10、11、13 日存在 `2 / 0 / 0` 条新增线索，并断言 7 月 10 日存在 `1` 条有效线索。这些数据在功能开发当天存在，但生产工作簿会被定时更新覆盖，后续可能暂时不包含新探陆。

当工作簿没有新探陆行时，产品要求是保留新探陆空区块并显示 `-`，但上述测试会错误失败，导致 GitHub Pages build job 中止并跳过 deploy。

## 修复策略

把“业务逻辑验证”和“当前生产数据状态验证”分离：

- 真实工作簿构建测试只验证新探陆 section 始终存在、顺序稳定、无目标时显示 `-`，不要求当前必须有新探陆实绩。
- 使用固定合成数据验证新探陆实际值、每日简报、NEV 总盘排除和全车系有效线索包含逻辑。
- 合成数据通过 mock `load_nev_daily()` 注入，不新增或维护完整二进制 Excel fixture。

## 合成数据

测试使用固定报告日期和稳定数据：

- NX8：新增线索 `10`、有效线索 `7`。
- 新探陆：新增线索 `2`、有效线索 `1`、新增到店 `0`。
- 新探陆目标为空。

分别构建“包含新探陆”和“不包含新探陆”的 payload，用两份结果的差值验证：

- 新探陆区块累计新增线索为 `2`。
- 每日简报存在独立新探陆板块且累计实绩为 `2`。
- NEV 总盘在两份 payload 中保持一致，证明新探陆未被计入四车总盘。
- 全车系有效线索在包含新探陆时增加 `1`，证明新探陆被纳入全车系口径。

## 数据产物

使用当前仓库源工作簿重新生成 `dashboard.json`、summary 和 7 月归档，使提交产物与源 Excel 保持一致。若当前源工作簿没有新探陆数据，新探陆区块应正常存在，实际值按缺失规则显示，而不是保留开发当天的瞬时值。

## 验证

1. 在主工作区运行 builder 专项测试和全量测试。
2. 创建带 `.git` 元数据的干净 Git worktree，避免发布测试把临时目录误判为非仓库。
3. 在干净 worktree 中运行 workflow 对应的 `python -m unittest discover -s tests -v`。
4. 运行 workflow 对应的保留时间构建命令，确认构建成功。
5. 推送修复并核对新的 GitHub Actions run 成功完成 build 与 deploy。

## 范围

- 不改变新探陆页面、汇总或简报业务逻辑。
- 不要求生产工作簿当前必须存在新探陆数据。
- 不降低 NEV 总盘和全车系有效线索的口径测试覆盖。
- 不新增 Excel 测试文件。
